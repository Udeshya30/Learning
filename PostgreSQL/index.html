1. What is PostgreSQL?

PostgreSQL is an advanced open-source relational database system known for:

  • ACID compliance
  • MVCC concurrency control
  • Advanced indexing
  • JSON support
  • Replication and HA
  • Extensibility
  • Transactional + analytical workloads
PostgreSQL today is not just a relational database. It is commonly used as: vector database, event store, analytics engine, AI metadata storage, queue system, and geospatial platform.

2. PostgreSQL Architecture

Core Components

  • Postmaster → main coordinator process
  • WAL Writer → durability logs
  • Checkpointer → flush checkpoints
  • Background Writer → writes dirty pages
  • Autovacuum → dead tuple cleanup
  • Shared Buffers → memory cache
  • WAL → write ahead logging

MVCC (Multi-Version Concurrency Control)

PostgreSQL allows readers and writers to work concurrently without blocking each other.

PostgreSQL creates row versions instead of overwriting rows directly.
Without VACUUM, dead tuples accumulate and table bloat becomes severe.

4. Schema Design

Normalization

  • 1NF → atomic values
  • 2NF → remove partial dependency
  • 3NF → remove transitive dependency

Primary Key Strategies

Type Pros Cons

UUID Distributed systems Index fragmentation
ULID Ordered UUID-like IDs Less common
Premature denormalization creates synchronization complexity.

5. SQL Mastery

Joins

  • INNER JOIN
  • LEFT JOIN
  • RIGHT JOIN
  • FULL OUTER JOIN
  • CROSS JOIN

Window Functions

SELECT
  user_id,
  amount,
  RANK() OVER (
    PARTITION BY user_id
    ORDER BY amount DESC
  )
FROM orders;

CTEs

WITH recent_orders AS (
  SELECT *
  FROM orders
  WHERE created_at > now() - interval '7 days'
)
SELECT count(*)
FROM recent_orders;

6. Transactions and Concurrency

ACID

  • Atomicity
  • Consistency
  • Isolation
  • Durability

Isolation Levels

  • Read Committed
  • Repeatable Read
  • Serializable
BEGIN;

UPDATE accounts
SET balance = balance - 100
WHERE id = 1;

UPDATE accounts
SET balance = balance + 100
WHERE id = 2;

COMMIT;

Deadlocks

Always lock rows in consistent order.

7. Indexing Deep Dive

B-Tree Index

CREATE INDEX idx_users_email
ON users(email);

GIN Index

CREATE INDEX idx_products_metadata
ON products USING GIN(metadata);

Partial Index

CREATE INDEX idx_active_users
ON users(status)
WHERE status = 'active';

Composite Index

CREATE INDEX idx_orders_user_created
ON orders(user_id, created_at DESC);
Indexes improve reads but slow down writes.
